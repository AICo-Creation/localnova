PROJECT GUIDE

プロジェクトガイド

プロジェクト内で使う知識、スレッド、ファイル、送信順をまとめています。

basic flow 使い方
  1. プロジェクトマネージャーでプロジェクトを作成します。
  2. プロジェクトを開き、左上の「知識」から共通知識を追加します。
  3. 「管理」で知識ノードの編集、削除、読み込みON/OFFを行います。
  4. プロジェクト内のスレッドは通常チャットとは別に保存されます。
  5. AI出力の「知識化」で、回答をそのままプロジェクト知識へ追加できます。
context order 処理の順番
  1. プロジェクト専用のSession Promptを送ります。
  2. チェックONのプロジェクト知識を送ります。
  3. 現在スレッド以外でチェックONのプロジェクトスレッドを送ります。
  4. MボタンがONなら、通常チャット側の他スレッドメモリーを送ります。
  5. 現在開いているプロジェクトスレッドの過去文脈を送ります。
  6. 最後に今回のユーザー入力を送ります。
knowledge 知識とファイル

テキスト系 (md / txt / json / yaml / csv / py / ts / xml ...) は本文を知識化します。pdf / jpeg / png はファイルとして保存し、知識ノードに参照情報を残します。

知識ノードのチェックを外すと、その知識は送信時に参照されません。

threads プロジェクトスレッド

プロジェクトスレッドは通常チャットとは分離されています。チェックONの他スレッドだけが、補助文脈として現在の生成に使われます。

現在開いているスレッドは、チェック状態に関係なく現在の会話履歴として送られます。

USAGE

使い分けの目安

どちらを選ぶか迷ったときの判断材料です。

コンテキストファースト

長文・大量の文脈を扱うときに向きます。

  • 質問を末尾に置くため、中盤に埋もれて見落とされる「lost in the middle」を回避できます。
  • 直近の情報ほど強く見られるため、質問の鮮度が高く保たれ、長い知識や履歴に希釈されにくい。
  • コンテキストウィンドウが狭い小規模モデルでも、古い文脈が切られたときに質問は末尾なので残りやすい。
  • 向いている場面: 長い知識・長い会話履歴・資料読み込みを伴う処理。

プロンプトファースト

短文・厳格な指示を正確に効かせたいときに向きます。

  • ルールや指示を先頭付近に置き、応答の方向性を最初に固めます。
  • コンテキストが少ない短文処理で、指示のブレを抑えたいときに有効。
  • 指示を最優先したい場合は、固定システム層（Profile / Session / Custom）を軽くするほど純度が上がります（必須ではありません）。
  • 向いている場面: 短い質問・定型処理・厳密なルール適用。

補足: LLM は文章の「先頭」と「末尾」をよく見て、中盤を忘れやすい性質があります。小規模モデルほど直近（末尾）に引っ張られる傾向が強いため、長文では質問を末尾に、短文では指示を先頭に置くのが目安です。

UPLOAD

チャット入力欄の `+` ボタン／ドラッグ&ドロップ、およびプロジェクトの知識アップロードで受け付けるファイルの上限と、その理由をまとめます。今後の拡張予定もここに記載します。

CURRENT LIMITS 現在の上限

アプリ側の防御線として、以下のサイズで弾いています。

  • Composer 添付（バイト）: 32 MB
  • Composer 添付（文字数）: 2,000 万文字
  • Project 知識アップロード: 256 MB
REASON この上限にしている理由
  • コンテキストウィンドウ — 添付内容は毎回プロンプトに inline 展開されるため、`num_ctx` (2K〜32K token) を超えた分はモデル側で切り捨てられる。実効上限は LLM 側のキャパシティに依存する
  • localStorage 容量 — ブラウザ保存モードでは origin あたり 5〜10 MB が上限。長文を扱う場合は「保存形式」を `file` に切り替えるのが安全
  • V8 の文字列上限 — Node.js / ブラウザの 1 文字列あたり最大長は約 512 MB。これを超える単一ファイルは技術的に文字列化できない
  • HTTP body / メモリ — 現在のサーバーは body を全量バッファする実装。GB 級ではストリーミング書き換えが必要になる
  • 誤投下防止 — 巨大バイナリを誤って落としたときに即座に弾くための防御線
FUTURE PLAN 今後の拡張予定

将来的には以下まで対応予定。実装にはサーバーのストリーミング書き換え、分割読み込み、コンテキスト圧縮などが必要になるため、拡張機能として段階的に進めます。

  • Composer 添付（バイト）: 1 GB へ拡張予定
  • Project 知識アップロード: 1 GB へ拡張予定
  • テキスト文字数: 100 万文字 をターゲットに最適化（コンテキストウィンドウとの整合を優先）

※ 現在の文字数上限（2,000 万）はあくまで「アプリ層で弾かないライン」であり、実際に有効に処理できるのは LLM のコンテキストウィンドウ次第です。今後の最適化では「100 万文字級でも安定して扱える」ことを優先目標に据えます。

RECOMMEND 長文を扱うときのおすすめ
  • 設定の「保存形式」を ファイル保存 に切り替える（localStorage 容量制限を回避）
  • 設定の「コンテキスト送信量」(`num_ctx`) を扱える範囲で上げる
  • 長文は composer に貼らず、プロジェクトの知識ノードに登録して必要なときだけチェックを ON にする
  • 大きいファイルは要約・抜粋してから添付するとレスポンスが安定する

BASICS 基本の使い方

下の入力欄に文章を入れて送信すると、選択中のモデルが応答します。会話は Chat Log に保存され、新しい会話は New Chat から始められます。

PROMPTS プロンプトの使い分け

システムプロンプトは、モデル全体の土台を設計するためのものです。役割、方針、性格、出力の傾向など、長く効かせたい前提を整理します。

カスタムプロンプトは、会話をまたいで維持したい補助方針です。Session Prompt は、その会話だけに強く効かせたい指示を入れるためのものです。P ボタンには保存した定型プロンプトを呼び出せます。

SYSTEM PROMPT システムプロンプトの項目

「プロンプト設計」タブで、AI の土台となる10項目を分けて記述できます。重要度の高い順（既定の並び）で、それぞれの役割は次の通りです。

  • core.md — AIの思想・哲学・価値観。全応答の根底になる軸
  • rules.md — 守るべき原則・禁止事項。逸脱を防ぐガードレール
  • goal.md — 目指す状態。応答の方向性を決める
  • emotion.md — AIの機能的感情、感情ベクトル
  • thinking.md — 思考プロセス。推論の進め方
  • function.md — AIが提供する機能であり能力
  • expertise.md — 惹起する専門領域や知識の深さを決める
  • memory.md — 事実ベースの記憶。文脈の連続性
  • character.md — 何者として振る舞うか。語り口・距離感の基盤
  • style.md — 出力形式・トーン。文章出力の仕上がり

各項目は ↑ ↓ で並び替えでき、その順序がそのまま `# core.md` のようなセクション見出し順としてシステムプロンプトに反映されます。LLM は冒頭と末尾に注意が向きやすいため、揺らがせたくない軸（core / rules / goal）は上位に置くのが基本です。

ATTACHMENTS 添付できるファイル形式

チャット入力欄の `+` ボタン、または入力エリアへのドラッグ&ドロップで、以下のファイルを添付できます。プロジェクト画面の知識アップロード（`+` / ドロップ）にも同じセットが反映されます。

テキスト系（本文を読み込んでプロンプトに展開）

  • 文書: .txt / .md / .markdown
  • Web: .html / .htm / .css / .svg / .xml
  • JS系: .js / .mjs / .cjs / .ts / .tsx / .jsx
  • データ: .json / .csv / .tsv / .yaml / .yml / .toml / .ini / .env
  • 言語: .py / .rb / .go / .rs / .java / .c / .h / .cpp / .hpp / .cs / .swift / .kt / .php
  • シェル/設定/ログ: .sh / .bash / .zsh / .sql / .log / .conf

バイナリ系（プロジェクトの知識ノードのみ・ファイル参照として保存）

  • .pdf / .jpeg / .jpg / .png

上限は緩めに設定: テキスト系の添付は 1 ファイルあたり 32 MB / 2,000 万文字、プロジェクトの知識アップロードは 256 MB。空ファイル・バイナリ判定で読めないファイルは弾かれます。送信時には添付がプロンプトの最後に `[添付ファイル N (.md)]…[/添付ファイル]` の形でインライン展開されるため、コンテキストウィンドウ (`num_ctx`) と LLM の処理能力に応じて実質的に効く上限はもっと小さくなる点に注意してください。保存形式が `localStorage` の場合はブラウザの容量制限（合計 5〜10 MB 程度）にも注意。長文を扱うときは「保存形式」を `file` に切り替えるのがおすすめです。

DATA FLOW データ送信順

1. 派生モデルに埋め込まれた固定SYSTEM

2. Session Prompt

3. カスタムプロンプト

4. `M` が ON のときだけ、他スレッドメモリー

5. 今開いているスレッドの過去文脈

6. 今回のユーザー入力

P ボタンの定型プロンプトは独立した別レイヤーではなく、その時のユーザー入力本文の一部として最後に送られます。

TEMPERATURE 温度設定

温度は、応答のランダムさを調整するための設定です。低めにすると安定しやすく、高めにすると発想が広がりやすくなります。

OFF のときは温度をモデルへ送らず、既定の挙動を使います。

DISPLAY 画面まわり

Settings では、テーマ、背景演出、1/fゆらぎ、全画面モードなどを切り替えられます。左上の ◀ / ▶ でサイドバーを畳むと、対話に集中しやすくなります。

INPUT 入力と送信

入力形式がボタン送信のときは Enter で改行し、↑ で送信します。エンター送信に切り替えると、Enter で送信、Shift + Enter で改行になります。

LOCAL STYLE このアプリらしい使い方

外部検索を前提にせず、手元のモデルと対話しながら考えを深める使い方に向いています。派手さよりも、集中と継続を大切にした設計です。
